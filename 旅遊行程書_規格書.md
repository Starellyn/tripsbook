# 旅遊行程書 — 模版規格書

> **給 AI 助手的使用說明**
> 你正在協助規劃一份旅遊行程書。請先完整讀這份規格，然後遵守：
> 1. 一律以 `旅遊行程書_模版.html` 為基礎產出新書，不要從零另寫版面。
> 2. 嚴守資料夾命名規則 `地點-YYYYMM`（見第十四章），檔名一律 `index.html`。
> 3. 所有會變動的內容只改 `TRIP` 資料區（`meta` / `days` / `food` / `pending` / `setup`；`meta` 內含 `phase`），不動版面與渲染程式。
> 4. 草稿期把未定選項放 `pending[]`；定案後清空它即成正式版。
> 5. 依旅程階段設 `meta.phase`（draft／final／travelling）：草稿期記帳/分帳/日記工具會自動鎖住，定案後才解鎖（見第二十二章）。
> 6. 涉及現況的資訊（花火日期、紅葉時機、價格、營業時間等）必須查證，不可憑印象填。
> 7. 產出後在書架 `index.html` 的 `BOOKS` 加一張卡，並提醒使用者發佈流程（第十二章）。

這份文件定義「旅遊行程書」HTML 模版的結構、資料欄位與設計規範。
目的：日後規劃任何一趟旅行時，照這份規格填資料，就能產出一致風格、可給家人討論、可連 Google Maps 的視覺化行程書。

配套檔案：
- `旅遊行程書_模版.html` — 空白模版（含範例資料與註解，照著改）
- `大阪旅遊行程書.html` — 已填好的完整範例（看「成品長怎樣」）
- `東京輕井澤草津行程書.html` — 含「待定／候選比較」「人數切換」「phase」的完整範例
- `index.html` — 書架首頁（所有旅遊書的總目錄，放 repo 根目錄）
- `_shared_共用資源.html` — 共用資源頁（每趟通用：我的地圖、App、行李清單、緊急聯絡）
- `_kakeibo_記帳.html` — 簡易記帳頁（單人記出入）
- `_warikan_分帳.html` — 多人分帳頁（一人記帳、自動結算）
- `_nikki_旅途日記.html` — 旅途日記頁（記實際 vs 計劃）
- `Google同步_AppsScript程式碼.gs` — 記帳/日記同步 Google 試算表的後端程式
- `Google同步_設定說明.md` — 同步的逐步設定教學
- `規劃準則.md` — 排行程的「內容面」準則（角色、品質要求、時令意識）
- `CLAUDE.md` — 給 AI 的專案總導覽（引用通用準則與本規格、規劃準則）

## 一、核心理念

三個不可動搖的設計目標，任何修改都不該違背：

1. **給家人討論用** — 打開即見全局（總覽頁），看點清楚，不需要技術背景就能讀。
2. **現場帶著用** — 每個地點、每一天都能一鍵開 Google Maps；離線（出國斷網）也要能正常顯示。
3. **改內容不改版面** — 所有會變動的文字集中在檔案最上方的 `TRIP` 資料區，改文字即可，版面自動生成。

---

## 二、檔案結構（單一 HTML，免安裝）

整份是一個 `.html` 檔，分三段：

| 段落 | 位置 | 是否常改 |
|---|---|---|
| `<style>` 設計規範 | 檔案上半 | 幾乎不動（改風格才動） |
| 版面骨架（封面／導覽／各分頁容器） | 中段 | 不動 |
| `const TRIP = {…}` 資料區 | `<script>` 開頭 | **每趟都改這裡** |
| 渲染程式 | 資料區下方 | 不動 |

設計刻意讓「資料」與「版面」分離：填行程的人只碰 `TRIP`，不必懂 HTML/CSS。

---

## 三、分頁架構

頂端導覽列固定為：

```
🏠 總覽 ‧ Day 1 … Day N ┃ 💰 預算 ‧ 🍴 美食 ‧ 🤔 待定
```

- **總覽**：每天一張卡片（日期＋主題＋區域＋當日預估），點卡片跳到該天。預設首頁。
- **Day N**：當天時間軸。由 `TRIP.days` 自動生成，有幾天就有幾顆按鈕。
- **預算**：固定支出表 ＋ 每日浮動三版本 ＋ 全程總計 ＋ 我的地圖／飯店盒。靜態 HTML，直接改文字。
- **美食**：整趟推薦餐廳卡片，由 `TRIP.food` 生成。
- **待定**：候選比較（草稿期用），由 `TRIP.pending` 生成；**清空資料即自動隱藏**（見第十節）。

切換為純前端顯示／隱藏，無外部依賴。

### 分頁以外的入口（連到外部工具頁）

除了上述分頁，還有幾個「按鈕」連到獨立工具頁（不是分頁，是 GitHub Pages 上的其他資料夾）：

- **預算分頁的工具盒**：🗺️ 我的地圖、🧾 旅遊記帳（`_kakeibo`）、👥 多人分帳（`_warikan`）、📔 旅途日記（`_nikki`）。後三者受 `phase` 控制，草稿期鎖住（見第二十二章）。
- **總覽頁底部**：💬 家人意見（連 Google 表單，見第十七章）。

這些工具頁各自獨立，詳見第十七～二十一章。

## 四、資料欄位定義

### 4-1　`TRIP.meta`（旅程基本資料）

封面、總覽底部、頁尾、頁面標題都讀這裡。

| 欄位 | 用途 | 範例 |
|---|---|---|
| `stamp` | 封面朱紅印章字（1–2 字） | `"大阪"` |
| `vlabel` | 封面右側直書小標 | `"大阪・京都・宇治の旅"` |
| `titleMain` | 主標前段（墨黑） | `"大阪・關西"` |
| `titleAccent` | 主標強調段（朱紅） | `"六日"` |
| `titleTail` | 主標尾 | `"旅遊行程書"` |
| `sub` | 副標（英文／羅馬字，可留白） | `"OSAKA · KYOTO"` |
| `chips[]` | 封面標籤，任意數量 | `["📅 9/25–9/30","🌙 5 晚 6 天"]` |
| `footStats[]` | 總覽頁底部統計，`{label,value}` | `{label:"全程",value:"5 晚 6 天"}` |
| `footer` | 頁尾文字 | `"祝旅途愉快 🧳"` |

### 4-2　`TRIP.days[]`（每日行程）

每個物件 = 一天。複製整個 `{…}` 即可新增一天。

**日層級欄位：**

| 欄位 | 用途 | 必填 |
|---|---|---|
| `tab` | 導覽列按鈕文字（如 `"Day 1"`） | ✔ |
| `num` | 頁內大標序號（如 `"DAY ONE"`） | ✔ |
| `title` | 當天主題標題 | ✔ |
| `date` | 日期（如 `"9/25 (四)"`） | ✔ |
| `theme` | 一句話動線 | ✔ |
| `area` | 每日小計：主要區域 | 建議 |
| `cost` | 每日小計：當日預估花費 | 建議 |
| `walk` | 每日小計：步調描述 | 建議 |
| `dayMap` | 「今日地圖」按鈕指向的地名 | 建議 |
| `items[]` | 當天每個行程點 | ✔ |

**行程點欄位（`items[]` 內每個物件）：**

| 欄位 | 用途 | 備註 |
|---|---|---|
| `time` | 時段（如 `"09:00"`、`"住宿"`） | 必填 |
| `type` | 類別，決定標籤顏色 | 見下方類別表 |
| `badge` | 標籤文字（自訂，如「景點」「午餐」） | 必填 |
| `title` | 地點／活動名稱 | 必填 |
| `desc` | 說明（看點、價位、注意事項） | 可留白 |
| `duration` | 建議停留時長 | 可留白 |
| `best` | 最佳造訪時段 | 可留白 |
| `transit` | 交通細節（多用於 `move` 類） | 可留白 |
| `note` | 紅框「顧問提醒」（預約／排隊／替代方案） | 可留白 |
| `map` | 地圖地名 → 有填才出現「📍地圖」按鈕 | 可留白 |
| `link` | 介紹網址 → 有填才出現「🔗介紹」按鈕 | 可留白 |

**`type` 類別與配色：**

| type | 標籤色 | 適用 |
|---|---|---|
| `move` | 藍 | 交通、移動 |
| `spot` | 金 | 景點、參觀 |
| `food` | 朱紅 | 餐廳、甜點 |
| `shop` | 抹茶綠 | 購物、商圈 |
| `hotel` | 灰 | 住宿 |

> 留白原則：任何可留白欄位若不填，該元素（按鈕／提醒框）就不會出現，不會留下空殼。

### 4-3　`TRIP.food[]`（美食分頁）

與每日行程獨立，放整趟想推薦的店。

| 欄位 | 用途 |
|---|---|
| `name` | 店名（同時用來生成地圖連結） |
| `price` | 單人價位帶 |
| `desc` | 招牌與特色一句話 |

## 五、Google Maps 連結規則

- 地圖按鈕用「地名搜尋」而非寫死座標，網址格式：
  `https://www.google.com/maps/search/?api=1&query=<地名>`
- 因此 `map`／`dayMap` 欄位**填正確、夠精確的地名**即可（建議含區域，如「鳥貴族 天神橋筋六丁目店」而非只寫「鳥貴族」）。
- 手機點擊會自動喚起 Google Maps App。
- 「我的地圖」按鈕（預算頁）需手動把分享連結貼進 `href`，這是整趟景點彙整、現場找路用。

**工作流程銜接**（沿用既有習慣）：
Excel 共享討論 → 定案 → 存進 Google 我的地圖 → 我的地圖連結貼進預算頁按鈕 → 行程書同時提供單點地圖與全程地圖。

## 六、設計規範（風格 token）

定義在 `:root`，改一處全站連動。

| 變數 | 色 | 用途 |
|---|---|---|
| `--paper` | 生成色紙底 `#F2EBDD` | 背景 |
| `--card` | 米白 `#FBF7EE` | 卡片底 |
| `--ink` | 墨黑 `#2A241E` | 主文字 |
| `--vermilion` | 朱紅 `#C2402A` | 重點、按鈕、強調 |
| `--indigo` | 藍 `#2E4A6B` | 交通資訊 |
| `--matcha` | 抹茶綠 `#6E8049` | 點綴 |
| `--gold` | 金 `#B08D3C` | 景點標籤 |

**字型**：使用系統內建字型堆疊（`--f-sans` / `--f-serif`），**不連外部 Google Fonts**。
理由：出國漫遊斷網時不掉字、開啟更快。蘋果走 PingFang／Songti，Windows 走微軟正黑，日文裝置走 Hiragino／Yu Mincho。

**版式原則**：襯線體做標題（旅誌感），無襯線體做內文（易讀）；卡片圓角＋柔和陰影；朱紅作為唯一的強調色，不濫用。

---

## 七、調整風格時的注意事項

- 改配色：只動 `:root` 變數，不要在個別元素硬寫顏色。
- 改字型：只動 `--f-sans` / `--f-serif`；若要換特殊字型，務必確認離線可用，否則違背「現場斷網」目標。
- 想做主題化（如可愛粉嫩風）：調 `--paper`／`--card`／`--vermilion` 三色即可換氣質，不必改結構。

---

## 八、刻意不做的事（避免過度設計）

以下功能評估後不納入，理由是維護成本高、實際使用率低、或讓檔案變肥：

- 深色模式切換、進場動畫 — 好看但用不到，徒增維護。
- 把預算做成可互動試算 — 預算討論回 Excel 更合適，行程書只需呈現結論。
- 連動外部 API（天氣、即時票價）— 違背「離線可用」，且資料易失效。

需要時再個案討論加入，預設保持精簡。

## 九、新行程的建立流程

1. 複製 `旅遊行程書_模版.html`，改名 `index.html`，放進新資料夾（命名見第十四章，如 `kyoto-202703/`）。
2. 改 `TRIP.meta`：封面、標籤、總覽底部、頁尾；並設 `phase:"draft"`（第二十二章）。
3. 設 `TRIP.setup`：人數／行李／包車分攤（第十一章），不需要就刪掉。
4. 依天數增減 `TRIP.days[]`，逐天填 `items`。
5. 填 `TRIP.food[]` 推薦餐廳。
6. 草稿期把還沒決定的選項放進 `TRIP.pending[]`（第十節）。
7. 改「預算」分頁的靜態表格文字（此頁不走資料區）。
8. 用瀏覽器打開檢查，發佈後丟群組給家人討論（第十二章）；可加家人意見表單（第十七章）。
9. **定案後**：清空 `pending`、`phase` 改 `final`、存 Google 我的地圖並把連結貼回預算頁按鈕。
10. **出發時**：`phase` 改 `travelling`，解鎖記帳／分帳／旅途日記（第十八～二十二章）。

> 工具頁（`_kakeibo`／`_warikan`／`_nikki`／`_shared`）是整個 repo 共用的，第一次建立後各趟沿用，不必每趟重做。

## 十、草稿版 → 正式版（同一份檔案走完整個生命週期）

這份模版刻意設計成**一份檔案涵蓋整段旅程**，不需要維護多個檔。完整的階段由 `phase` 控制（draft／final／travelling，見第二十二章）；本章聚焦「草稿 → 正式」這個內容上的轉變，對應 `phase` 從 `draft` 到 `final`。

### 草稿版 vs 正式版的內容差異

| | 草稿版（phase: draft） | 正式版（phase: final / travelling） |
|---|---|---|
| 目的 | 跟家人討論、做決定 | 定案後現場執行 |
| 候選 | `pending[]` 並列多個選項 | 已清空，選中的搬進 `days[]` |
| 花費 | 區間（省錢／舒適／享受） | 可改為實際已訂金額 |
| 記帳/日記工具 | 鎖住（見第二十二章） | 解鎖 |
| 心態 | 還在喬 | 照這個走 |

> 出發後還有 `travelling` 階段（旅途中），與 `final` 的內容相同，差別只在記錄工具的使用，詳見第二十二章。

### `TRIP.pending[]`（候選比較）欄位

每個物件 = 一個待決定的主題（如「東京住宿」）。

| 欄位 | 用途 |
|---|---|
| `topic` | 待決定主題標題 |
| `note` | 一句話補充（可留白） |
| `options[]` | 候選清單，每個 `{name, price, status, pick, desc, pros[], cons[], map, link}` |

候選 `options` 內欄位：

| 欄位 | 用途 |
|---|---|
| `name` | 選項名稱 |
| `price` | 價位 |
| `status` | `"待決定"`／`"傾向"`／`"已排除"`，顯示為不同顏色標籤 |
| `pick` | `true` 時把卡片框成朱紅、加 ✔（標記目前首選） |
| `desc` | 一句話定位 |
| `pros[]` | 優點清單（綠色 ＋ 號） |
| `cons[]` | 缺點清單（紅色 － 號） |
| `map` / `link` | 同行程點，產生地圖／介紹按鈕 |

### 自動切換機制

- 只要 `TRIP.pending[]` 有內容，導覽列就出現「🤔 待定」分頁。
- **全部決定後，把 `pending` 清空（`pending:[]`）或刪掉整個欄位** → 「待定」分頁自動消失，同一份檔案就從草稿版變成正式版。
- 切換邏輯靠 JS 自動偵測，不需手動改版面。

## 十一、基本設定：人數／行李切換器

規劃的**第一步**應該先確認人數與行李，因為它是上游決策——人數一變，包車車型、分攤、住宿房型全部跟著變。

### 為什麼放在最前面

| 影響項目 | 少人數 | 多人數 |
|---|---|---|
| 包車分攤 | 每人偏貴 | 同車分更多份，每人變便宜 |
| 車型 | 一般車／Alphard | 行李多需 Jumbo／Hiace 大車 |
| 房型 | 一間房 | 多間房或找大房型 |

行李數決定車型：4 人＋4 大行李箱，普通車塞不下，得指定大型車，報價也不同。

### 位置與運作

- 顯示在**總覽頁最上方**的「👥 基本設定」區塊。
- 提供人數切換按鈕（如 2 人／4 人），點選後**即時重算**包車每人分攤、住宿方向、行李對應車型。
- 純前端切換，不需重新整理。

### `TRIP.setup` 欄位

| 欄位 | 用途 |
|---|---|
| `default` | 預設顯示的人數 |
| `options[]` | 各人數情境，每個含下列欄位 |

每個 option：

| 欄位 | 用途 |
|---|---|
| `people` / `label` | 人數與按鈕文字 |
| `charterTotal` | 包車全程一口價（整台車總價，台幣）；無包車填 0 |
| `car` / `carNote` | 建議車型與說明 |
| `luggage` | 行李量描述 |
| `room` / `roomEach` / `roomNote` | 住宿房型、每人分攤、說明 |

每人包車分攤由 JS 自動算（`charterTotal ÷ people`），不必手填。

### 不需要時

刪掉 `TRIP.setup` 整個物件，總覽頁的設定區塊會自動移除（適合純市區、不包車的行程）。

## 十二、發佈到網路（GitHub Pages）

行程書是純靜態 HTML（無後端、字型離線），可直接用 GitHub Pages 免費代管，家人點連結就能看完整可互動的網頁。

### 為什麼不是 Google Drive

Google Drive 是「雲端硬碟」，只負責保管檔案——別人點連結是下載檔案或看到原始碼，**不會把 HTML 跑成網頁**（Google 2016 年已關閉代管功能）。GitHub Pages 才是「網站代管」，把 HTML 當網頁跑給人看。一句話：Drive 是倉庫，GitHub Pages 是櫥窗。

### 一次性設定（已完成，記錄備查）

1. 建一個 repository（例：`tripsbook`），設 Public。
2. Settings → Pages → Source 選 `Deploy from a branch` → Branch `main`、資料夾 `/ (root)` → Save。
3. 約 1 分鐘後生效，根網址為 `https://<帳號>.github.io/<repo>/`。

本專案實例：帳號 `Starellyn`、repo `tripsbook`、根網址 `https://starellyn.github.io/tripsbook/`。

### 目錄結構（一個 repo 裝下所有旅遊書）

**不要一本一個 repo**。一個 repo 用資料夾分每趟，設定一次、長期沿用（資料夾命名見第十四章）：

```
tripsbook/
├── index.html              ← 書架首頁（總目錄，放根目錄）
├── tokyo-202610/
│   └── index.html          ← 東京輕井澤草津（地點-YYYYMM）
├── osaka-202504/
│   └── index.html          ← 大阪
├── _shared/index.html      ← 共用資源（每趟通用）
├── _kakeibo/index.html     ← 簡易記帳
├── _warikan/index.html     ← 多人分帳
└── _nikki/index.html       ← 旅途日記
```

對應網址：
- 書架總入口 → `https://<帳號>.github.io/<repo>/`（傳給家人的主連結）
- 各本 → `https://<帳號>.github.io/<repo>/<地點-YYYYMM>/`

### 兩條鐵則（踩雷點）

1. **每本的檔名一律用 `index.html`**：這樣網址停在資料夾（`/tokyo-202610/`）就能開，不必在後面加檔名。用中文原檔名會讓網址變成一長串編碼，又醜又難傳。
2. **網址一定要帶資料夾名**：`/<repo>/` 是根目錄，只有書架；各本行程書在 `/<repo>/<地點-YYYYMM>/`。少了資料夾名會 Page not found。

### 更新流程（VS Code）

```
VS Code 開 tripsbook 資料夾
├─ 改任一本的 index.html → 存檔
├─ Source Control 寫 commit 訊息 → ✓ → Sync/Push
└─ 等 1–2 分鐘，該網址自動更新（瀏覽器有快取時重新整理或用無痕視窗）
```

同一個網址永遠是最新版，不需重傳檔案或換連結——草稿改完 push、定版清掉待定再 push，連結始終不變。

## 十三、書架首頁（index.html）

所有旅遊書的總目錄，放 repo 根目錄。視覺與行程書同一套和風旅誌風格，資料驅動。

### 加一本書的流程

1. 複製模版 → 填內容 → 改名 `index.html` → 放進新資料夾（命名見第十四章，如 `kyoto-202703/`）上傳。
2. 開書架 `index.html`，在 `BOOKS` 陣列加一張卡（或把占位卡的 `url` 補上），並填 `sortKey`。
3. push → 書架自動多一本、自動依時間排序。

### `BOOKS[]` 欄位

| 欄位 | 用途 |
|---|---|
| `seal` | 書脊上的印章字（1–2 字，城市／國家） |
| `title` | 書名 |
| `dates` | 日期／副標 |
| `desc` | 一句話簡介 |
| `tags[]` | 主題標籤（emoji ＋ 文字） |
| `status` | `"草稿"`／`"定版"`／`"構想中"`／`"已成行"`，顯示為不同顏色標籤 |
| `url` | 該書資料夾相對路徑（結尾加 `/`）；**留空 `""` 則卡片顯示「尚未上線」、不可點** |
| `sortKey` | `YYYYMM` 數字，用來排序（最新在上）；占位卡填 `0` 自動沉底 |
| `c1` / `c2` | 書脊漸層兩色（不同書本不同封面感） |
| `soon` | `true` 顯示為虛線占位卡（「下一趟」用） |

### 卡片狀態（四種）

| status | 顏色 | 意義 |
|---|---|---|
| `構想中` | 藍 | 還沒開始細排 |
| `草稿` | 朱紅 | 規劃中、`pending` 還有候選 |
| `定版` | 抹茶綠 | 已定案、`pending` 清空 |
| `已成行` | 灰 | 旅程已結束，留作回憶錄 |

### 排序

書架以 `sortKey`（`YYYYMM`）**由大到小**排列，最新的旅程顯示在最上方；占位卡 `sortKey:0` 永遠沉底。出發前最常看的就是最近那趟，放最上最順手。

### 設計要點

- 用**相對路徑**（`tokyo-202610/` 而非完整網址），本機預覽、線上都能正確連到各本。
- 卡片狀態與行程書的「草稿／正式」概念一致：`草稿` 對應還有待定、`定版` 對應已清空待定。
- 加新書只動 `BOOKS` 資料，版面自動排列。

## 十四、資料夾命名原則

### 格式：`地點-YYYYMM`

- **地點**：英文小寫（網址乾淨；中文會變成一長串編碼）。
- **YYYYMM**：西元年＋月，四位年＋兩位月（如 `202610`）。

範例：
- `tokyo-202610` — 2026 年 10 月東京
- `osaka-202504` — 2025 年 4 月大阪
- `kyoto-202703` — 2027 年 3 月京都

### 為什麼這樣訂

| 考量 | 說明 |
|---|---|
| 用 `YYYYMM` 數字 | 檔案／網址自動按時間排序，一眼看出先後 |
| 地點用英文小寫 | 網址乾淨（`/tokyo-202610/`），易傳給家人 |
| 一年兩次同地點 | 年月不同自然分開（`tokyo-202604`、`tokyo-202610`） |
| 同月去兩地 | 地點不同不衝突（`tokyo-202610`、`osaka-202610`） |

### 規則重點

- 資料夾名 = `地點-YYYYMM`，裡面的檔案一律 `index.html`。
- 改了資料夾名，記得同步更新書架 `index.html` 對應卡片的 `url` 與 `sortKey`。

## 十六、未來可擴充（先知道有這選項）

## 十七、家人意見回饋（Google 表單按鈕）

純靜態網頁無法存「家人在頁上打的字」，所以用 Google 表單收集意見最務實。

### 設定步驟

1. 到 Google 表單（forms.google.com）建一份，欄位建議：哪一天／哪個項目、想怎麼調整、其他想法。
2. 表單右上「傳送」→ 取連結（可勾選縮短網址）。
3. 把行程書總覽頁那顆 `feedback-btn` 的 `href` 換成你的表單網址。

### 運作

- 家人點「💬 看完有想法？」→ 跳到表單填 → 意見全部進一份試算表，你一覽無遺。
- 不破壞純靜態架構，零成本。
- 一份表單可共用多趟，或每趟各建一份。

> 進階：可讓表單回應與記帳／日記匯到**同一份 Google 試算表**，集中查看，見第二十三章。

## 十八、旅遊記帳頁（_kakeibo）

單機記帳工具，記錄旅途收支，存在瀏覽器 localStorage。

### 檔案與位置

`_kakeibo_記帳.html`（上線時放 `_kakeibo/index.html`），行程書預算頁的工具盒有「🧾 旅遊記帳」按鈕連到 `../_kakeibo/`。

### 功能

- 記一筆：收入／支出切換、項目、金額（¥）、分類、日期。
- 自動加總收入／支出／結餘，結餘附台幣換算。
- 明細依日期分組、顯示當日支出小計。
- 匯出 CSV 備份、清空。

### 重要限制（務必讓使用者知道）

- 預設只存在**該手機的該瀏覽器**，不上傳；換手機、清除瀏覽器資料會消失 → **回程前按「匯出」備份**，或開啟 Google 同步（第二十章）作雲端備份。
- 適合「一個人用一支手機記」的情境；要多人同步或分帳，用多人分帳頁（第十九章）或現成 App（如 Splitwise）。

### 每趟分開帳本

程式頂端的 `KEY` 常數決定帳本歸屬（如 `tripsbook_expense_tokyo202610`）。**換一趟旅程要改這個 `KEY`**，帳本才不會混在一起。

### 為什麼這裡用 localStorage

行程書／書架本身**不靠 localStorage 存資料**——它們的內容寫死在 `TRIP`／`BOOKS`，互動（分頁、人數切換、phase）只是純前端顯示邏輯，重整後依資料重新生成即可。

但記帳是**使用者當場輸入、需要留存**的資料，沒有後端就得存在瀏覽器，因此用 localStorage 正好適用（放在自己的 GitHub Pages 網域可正常運作）。若要永久保存或多人查看，再另接 Google 試算表（第二十章）。

## 十九、多人分帳頁（_warikan）

與簡易記帳並存的進階版，給團體出遊用：一人記帳、自動算出誰該補誰。

### 檔案與位置

`_warikan_分帳.html`（上線時放 `_warikan/index.html`），行程書工具盒有「👥 多人分帳」按鈕連到 `../_warikan/`。

### 功能

- 設定**同行成員**（可增刪；已有帳目的成員不能刪）。
- 記一筆：項目、金額、分類、**誰付的**、**分攤給誰**（可全員均分或只選部分人）、日期。
- **自動結算**：算出每人淨額（該收回／該付出），並產生**最少筆數的清償建議**（如「B → 我 ¥5,000」）。
- 匯出 CSV（含結算建議）、清空帳目。

### 結算邏輯（已驗算）

每人「付出總額 − 應分攤總額」＝淨額；正數該收回、負數該付出。再用貪婪配對產生最少轉帳筆數。淨額總和恆為 0。

範例（4 人）：我付晚餐 12,000＋幫 BC 付門票 2,000、A 付車資 4,000、晚餐車資全員均分 → 結算：B→我 5,000、C→我 5,000。

### 能與不能

| 能做 | 不能做 |
|---|---|
| 一支手機記全團、標墊付與分攤、自動結算 | 各自手機即時同步（需後端） |

要各自手機同步或更複雜分帳，建議用現成 App（如 Splitwise）。

### 每趟分開

同樣靠頂端 `KEY` 常數（如 `tripsbook_split_tokyo202610`）區分帳本，換趟要改。

### 兩個記帳頁怎麼選

| | 簡易記帳 `_kakeibo` | 多人分帳 `_warikan` |
|---|---|---|
| 情境 | 自己隨手記出入 | 團體出遊、要算清償 |
| 重點 | 收支結餘 | 誰付誰分、誰補誰 |

## 二十、記帳同步到 Google 試算表

localStorage 怕清快取、換手機就沒了。可選擇把每筆額外寫進 Google 試算表（永久保存、可多人查看）。

### 架構

```
記帳頁（GitHub Pages）→ Apps Script 網址（小後端）→ Google 試算表
```

純靜態網頁不能直接寫 Google 試算表，需 Apps Script 當橋樑。**localStorage 仍是主儲存**（離線照記），Google 是額外備份——兩者並存，網路或後端失敗都不會丟資料。

### 配套檔案

- `Google同步_AppsScript程式碼.gs` — 貼進試算表 Apps Script 的後端程式
- `Google同步_設定說明.md` — 逐步設定教學

### 啟用方式

1. 依說明檔建試算表、貼程式、部署成網頁應用程式、取得網址。
2. 把記帳頁／分帳頁最上面的 `const SHEET_API=""` 填成那個網址。
3. 留空 `""` ＝ 只存本機、功能照常（向後相容）。

### 設計重點

- 送出用 `fetch` + `mode:"no-cors"`（瀏覽器限制，讀不到回應但寫入成功）。
- 頁面有同步狀態小標：同步中／已同步 ✓／同步失敗（已存本機）。
- 刪除不真的移除試算表的列，而是把「動作」欄標為「已刪除」，保留軌跡。
- 多趟可共用一份試算表：靠 `TRIP_NAME` 自動分頁，換趟改 `KEY` 與 `TRIP_NAME`、`SHEET_API` 共用。

### 隱私

- 「誰可以存取＝所有人」指知道網址者能送資料進來；網址不外流即可。
- 試算表本身的檢視權限另外管，建議分享給「特定人」而非「知道連結都能編輯」。

## 二十一、旅途日記（_nikki）：記錄實際 vs 計劃

行程書是「計劃版」，旅途中難免微調。旅途日記用來記**實際去了哪**，回顧時看到真實版本（計劃版本身也保留作對照）。

### 設計理念

- **不改計劃版行程書**：行程書留作「本來想怎麼走」，是回顧的一部分。
- 旅途中**手機一鍵記**實際狀況，符合旅途節奏。
- 計劃版 + 旅途日記 ＝ 完整回顧（「原本要去 A，臨時去了 B，超讚」）。

### 檔案與位置

`_nikki_旅途日記.html`（上線放 `_nikki/index.html`），行程書工具盒有「📔 旅途日記」按鈕。

### 功能

- 記一站：**與計劃的關係**（✅照計劃／🔄改去別處／➕臨時加碼／⏭️跳過）、實際地點、原計劃（改/跳時填）、日期、⭐評分、心得、照片/地圖連結。
- 四種關係用不同顏色左邊框與標籤區分。
- 依日期排序（舊到新，照旅程順序回顧）。
- localStorage 存，可選 Google 同步、可匯出 CSV。

### 與其他工具的關係

| 工具 | 角色 |
|---|---|
| 行程書 | 計劃版（出發前定案） |
| 旅途日記 | 實際版（旅途中記錄） |
| 記帳／分帳 | 實際花費 |

三者合起來＝出發前的計劃 + 旅途的真實 + 花費，構成完整旅程紀錄。

### Google 同步

同記帳機制，`SHEET_API` 填網址即啟用。Apps Script 已能辨識日記資料（有 `place` 欄），自動寫進「<帳本>-日記」分頁，與記帳分開。換趟改 `KEY` 與 `TRIP_NAME`。

## 二十二、旅程階段與工具解鎖（phase）

工具是跟著旅程時間軸走的——草稿期連去哪都沒定，記帳、記實際足跡都還沒發生。用 `phase` 讓工具隨階段解鎖，從介面引導正確順序。

### 三階段 × 該用什麼

| 階段 `phase` | 狀態 | 該用 | 不該用（鎖住） |
|---|---|---|---|
| `draft` | 草稿 | 行程、待定候選、人數試算、家人意見 | 記帳、分帳、旅途日記 |
| `final` | 定案 | 清空待定、確認住宿、存我的地圖 | （工具已開放，出發前備而不用） |
| `travelling` | 旅途中 | 記帳、分帳、旅途日記 | 大改行程結構（微調記日記即可） |

核心邏輯：記帳記「真實花費」、日記記「實際足跡」，兩者只在出發後存在；草稿期它們本該是空的。

### 運作

- 在 `TRIP.meta` 設 `phase:"draft"`（或 `final` / `travelling`）。
- `draft` 時，記帳／分帳／旅途日記按鈕**變灰、標「🔒 定案後開放」、點不了**；工具盒下方顯示階段提示。
- 切成 `final` 或 `travelling`，按鈕自動解鎖可點。
- 「我的地圖」按鈕不受限（各階段都可能用到）。

### 建議切換時機

1. 開始規劃 → `draft`（同時 `pending` 有候選）。
2. 住宿景點定案、清空 `pending` → 改 `final`（順手一起改）。
3. 出發當天 → 改 `travelling`，旅途中安心用三個記錄工具。

### 與其他機制的一致性

這延續「待定分頁沒資料就自動隱藏」的精神——介面只呈現當下階段該用的東西，避免用錯順序。`phase` 把這個邏輯擴大到所有工具。

## 二十三、資料匯流：所有 Google 資料集中一份試算表

這趟系統有「兩套 Google 通道」，刻意各用對的工具，但**資料匯到同一份試算表**，回顧時打開一份檔就看全部。

### 兩套通道（各司其職，不強行統一）

| 用途 | 工具 | 為什麼用這個 |
|---|---|---|
| 家人意見 | Google **表單** | 偶爾填、要結構化問題與防呆，表單自帶介面最適合 |
| 記帳／分帳／日記 | 自製頁 + **Apps Script** | 旅途中頻繁快速記、要即時漂亮介面 |

> 不把意見也改成自製留言框：那等於重做表單已內建的選項／必填／多筆管理，CP 值低。讓兩者各用所長，再把資料匯流即可。

### 整合做法：匯到同一份試算表

目標是「一份試算表、多個分頁」：

```
我的旅遊資料（一份 Google 試算表）
├── 東京202610          ← 記帳（Apps Script 自動建）
├── 東京202610-日記      ← 旅途日記（Apps Script 自動建）
└── 表單回應 1           ← 家人意見（表單連結到此）
```

設定方式：

1. **記帳／分帳／日記**：依第二十章設好 Apps Script，它的 `SHEET_API` 指向「我的旅遊資料」這份試算表。資料會自動依 `TRIP_NAME` 建分頁。
2. **家人意見表單**：在 Google 表單的「回應」分頁，點試算表圖示 → 選擇**連結到「我的旅遊資料」這份既有試算表**（而非另建新表）。表單回應就會以一個分頁存進同一份檔。
3. 結果：記帳、日記、意見全部落在同一份試算表的不同分頁，集中查看。

### 多趟的組織

- 同一份試算表可服務所有旅程：每趟記帳／日記靠 `TRIP_NAME` 自動分頁（如 `東京202610`、`大阪202504`）。
- 家人意見若每趟各建一份表單，回應分頁可都連到這份總表；或共用一份表單、用題目區分旅程。
- 換趟只改記帳頁的 `KEY`、`TRIP_NAME`，`SHEET_API` 共用。

### 為什麼這樣最划算

- **力氣最小**：不用多寫程式，表單回應改個「連結到既有試算表」的設定即可。
- **單一資料源**：回顧一趟旅程時，打開一份試算表就有意見、花費、實際足跡。
- **各用所長**：表單做問卷、自製頁做即時記錄，沒有勉強統一造成的副作用。

## 二十四、網站 ↔ Google Drive 對應

擁有者**早已有一套運作中的 Google Drive 整理習慣**：一趟一個資料夾，內含規劃試算表、pdf、我的地圖。網站系統**對齊**這套既有習慣，而非另立一套。

### 兩個世界、各保留習慣

| | Google Drive（既有） | 網站 GitHub Pages |
|---|---|---|
| 角色 | 資料中心：規劃試算表、pdf、我的地圖、記帳 | 對外展示（給家人看）＋ 旅途即時記錄 |
| 命名 | 民國年月　`民國YYMM-地點旅遊資料` | 西元年月　`地點-YYYYMM` |
| 結構 | 一趟一個資料夾，整包收一起 | 一趟一個資料夾，放 `index.html` |

> 兩者是同一趟的不同階段：Drive 試算表討論規劃 → 定案後做成網站行程書給家人看。

### 命名對照（民國 ↔ 西元）

換算規則：**民國年 + 1911 = 西元年**。

| Drive 資料夾（民國） | 網站資料夾（西元） | 說明 |
|---|---|---|
| `11211-東京旅遊資料` | `tokyo-202311` | 民國 112 年 11 月 = 2023/11 |
| `11305-大阪旅遊資料` | `osaka-202405` | 民國 113 年 5 月 = 2024/05 |
| `11409-大阪旅遊資料` | `osaka-202509` | 民國 114 年 9 月 = 2025/09 |
| `11502-東京旅遊資料` | `tokyo-202602` | 民國 115 年 2 月 = 2026/02 |

> 注意：本系統第一本 `tokyo-202610`（2026/10 東京輕井澤草津）對應民國 `11510`。

### 三個連接點

1. **我的地圖**：行程書「🗺️ 我的地圖」按鈕，直接貼該趟 Drive 資料夾裡那份 My Maps 的分享連結。
2. **記帳／分帳／日記**：Apps Script 的 `SHEET_API` 指向該趟 Drive 資料夾裡的**規劃試算表**，記帳資料成為它的新分頁（`TRIP_NAME` 對應分頁名）。整包收在一起，符合既有習慣。
3. **互相指向**：可在 Drive 試算表或資料夾放一個網站行程書連結；網站預算頁也連回我的地圖。兩邊互通。

### 建立新行程時的對應動作

1. Drive：照既有習慣建 `民國YYMM-地點旅遊資料` 資料夾（放規劃試算表、pdf、我的地圖）。
2. 網站：建 `地點-YYYYMM` 資料夾放 `index.html`（西元，查上方對照）。
3. 行程書按鈕填上 Drive 那份我的地圖連結；記帳 `SHEET_API` 指向那份規劃試算表。
4. 書架 `index.html` 加一張卡。

### 原則

網站**對齊** Drive 既有習慣，不另立命名系統。擁有者腦中的「資料中心」始終是 Drive 各趟資料夾；網站是它的對外展示與即時記錄延伸。
