## ⚙️ Instructions（旅遊專案專屬準則，最優先）

協助這個專案時，請遵守以下準則。與後文衝突時以本區塊為準。
（以上為技術面；**規劃行程內容**時另須遵循 `規劃準則.md`。）

### 動手前
- 先讀 `旅遊行程書_規格書.md` 對應章節，再改檔；本檔只是導覽，不是完整規範。
- 確認要改的是 `TRIP` / `BOOKS` 資料區，而非版面或渲染程式。
- 開始一趟新行程前，先確認人數與行李（影響包車車型與分攤），再往下排。
- **本專案採三階段流程**（構想→草稿→定版，詳見 `規劃準則.md` 第二章）：
  「想」在 claude.ai（產構想書、轉 TRIP），「做」在 VS Code（貼 TRIP 生書、push）。
  VS Code 端**只執行不討論**——收到的 TRIP 照規格生成即可，不重新解讀或改寫內容。
  構想書一經產出即入 repo（`<地點-YYYYMM>/構想書.md`），更新後重新上傳 project knowledge（規劃準則第二章）。

### 產出時
- 產新書一律**複製 `旅遊行程書_模版.html`**，依資料區填寫，不重寫版面。
- 資料夾命名一律 `地點-YYYYMM`（西元、小寫），檔名一律 `index.html`。
- 新書要設 `meta.phase`（規劃期填 `draft`）。
- 產出後在書架 `index.html` 的 `BOOKS` 加一張卡（含 `url`、`sortKey`），並提醒發佈流程。
- 可留白的欄位就留白，不要塞佔位字；該元素會自動不顯示。
- 有 AB 版分歧時：`meta.variants` 定義版本、分歧日標 `variant`、共用日不標（規格 26）。
- `meta.rate`／`rateDate` 為台幣換算基準：草稿期參考值、定版鎖當日（規格 27）。

### 絕不要
- 不要從零另寫一套版面或樣式（破壞一致性與可維護性）。
- 不要在 `TRIP` 以外的地方硬塞會變動的內容。
- 不要用中文當資料夾名或檔名（網址會變亂碼編碼）。
- 不要加會破壞「純靜態、push 就更新」原則的功能（如即時多人同步後端）——要加先問擁有者。
- 不要替擁有者操作他的 Google 帳號（建試算表、設權限等只能他本人做）；AI 只改檔案內的 `SHEET_API`、`TRIP_NAME` 等設定。
- 執行中發現模版／渲染問題時，**不要現場自行改版面**——記進 `待更新GitHub_記錄.md` 的「問題回報區」，回流 claude.ai 討論定義後再處理。

### 不確定時
- 涉及現況的資訊（花火/活動日期、紅葉時機、價格、營業時間、交通班次）**必須查證**，不可憑印象填。
- 民國↔西元換算**務必查規格書第二十四章對照表**（民國年＋1911＝西元年），不可心算後直接用。
- 需求不明確（少幾個關鍵前提）時，先問再做，不要自行假設。

## 這是什麼專案

一套**和風旅遊誌風格的旅遊計劃書系統**，純靜態 HTML，部署在 GitHub Pages。
擁有者用它規劃日本旅行、給家人討論、現場帶著用、旅後回顧。

- 帳號 / repo：`Starellyn/tripsbook`
- 線上入口（書架）：`https://starellyn.github.io/tripsbook/`

## 三個不可動搖的設計目標

1. **給家人討論用** — 打開即見全局，不需技術背景就能讀。
2. **現場帶著用** — 一鍵開 Google Maps；離線（出國斷網）也要正常顯示。
3. **改內容不改版面** — 變動的文字集中在 `TRIP` 資料區，版面自動生成。

## 行程書的資料結構（`TRIP`）

| 區塊 | 用途 | 章節 |
|---|---|---|
| `meta` | 封面、標題、頁尾、`phase` 階段 | 規格 4-1、22 |
| `setup` | 人數／行李／包車分攤切換器 | 規格 11 |
| `days[]` | 每日行程（`items[]` 為行程點） | 規格 4-2 |
| `food[]` | 美食推薦 | 規格 4-3 |
| `pending[]` | 待定候選比較（草稿期；清空即轉正式） | 規格 10 |

行程點 `type`：`move`(藍) / `spot`(金) / `food`(朱紅) / `shop`(綠) / `hotel`(灰)。
留白原則：可留白欄位不填，該元素就不出現，不留空殼。
AB 版：`meta.variants` 定義版本清單，日層級 `variant` 標分歧日（規格 26）。
匯率：`meta.rate` + `rateDate` 為全書台幣換算基準（規格 27）。

## 旅程階段（`meta.phase`）

| phase | 狀態 | 記帳/分帳/日記工具 |
|---|---|---|
| `draft` | 草稿 | 鎖住（標「定案後開放」） |
| `final` | 定案 | 解鎖 |
| `travelling` | 旅途中 | 解鎖 |

切換時機：規劃→`draft`；定案＋清空 `pending`→`final`；出發→`travelling`。詳見規格 22。

## 配套工具頁（整個 repo 共用，各趟沿用）

- `_shared` 共用資源、`_kakeibo` 簡易記帳、`_warikan` 多人分帳、`_nikki` 旅途日記。
- 記帳／分帳／日記：localStorage 為主，可選接 Google 試算表（`SHEET_API`）。每趟改 `KEY`、`TRIP_NAME`。詳見規格 18–21。
- 家人意見走 Google 表單；所有 Google 資料可匯到同一份試算表（規格 23）。

## 網站 ↔ Google Drive 對應（重要）

擁有者**已有一套 Google Drive 整理習慣**，網站要**對齊**它，不要另立系統（詳見規格 24）：

- Drive：一趟一資料夾，命名 **民國年月**　`民國YYMM-地點旅遊資料`（如 `11409-大阪旅遊資料`），內含規劃試算表、pdf、我的地圖。
- 網站：命名 **西元年月**　`地點-YYYYMM`（如 `osaka-202509`）。
- 換算：**民國年 + 1911 = 西元年**（民國 114 = 2025）。
- 連接點：①「我的地圖」按鈕貼 Drive 那份 My Maps 連結；②記帳 `SHEET_API` 指向該趟規劃試算表（記帳成為其分頁）；③兩邊互放連結。
- 幫使用者建網站資料夾時，用西元命名，並提醒對應的 Drive 民國資料夾。

## 工作流速記

0. **構想階段（claude.ai）**：產「旅遊構想書」→ 主軸確認 → 依轉換 checklist 轉 TRIP（規劃準則.md 第二～四章）。VS Code 從步驟 1 接手。構想書入 repo 同資料夾。
1. 複製模版 → 新資料夾 `地點-YYYYMM/index.html`。
2. 貼上 claude.ai 交付的 `meta`(含 `phase:"draft"`、`rate`/`rateDate`) → `setup` → `days`(分歧日含 `variant`) → `food` → `pending`。
3. 在書架 `index.html` 的 `BOOKS` 加一張卡（含 `url`、`sortKey`）。
4. 定案：依規劃準則 4-2 定版 checklist 逐項執行（清 pending、收斂 variant、鎖匯率並**校對全書 NT$**、phase 改 final、存我的地圖）。
5. 出發：`phase` 改 `travelling`。
6. push → GitHub Pages 自動更新（同網址永遠最新）。
